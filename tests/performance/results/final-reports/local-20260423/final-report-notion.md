# k6 데모앱 성능 테스트 최종보고서

> 핵심 결론  
> 이번 본실행에서 핵심 시나리오 5개는 모두 기준을 통과했고, 서버 장애(`5xx`)는 한 번도 발생하지 않았다. 추가 보완 실행에서도 상품 목록 조회 API는 자원 제한 스택 기준 `200 VUs`까지 안정적으로 동작했다.

## 한눈에 보기

- 작성일: 2026-04-23
- Run ID: 20260423-final-report
- 대상 프로젝트: k6 Pickup Market Demo
- 실행 환경: 자원 제한 Docker Compose 스택
- Base URL: http://localhost:14000
- 측정 방식: k6, Prometheus, Grafana

## 이번 테스트에서 확인한 것

1. 상품 목록 조회가 반복 부하에서 안정적으로 응답하는지 확인했다.
2. 상품 등록 burst 상황에서 `201`과 `409`가 의도대로 분리되는지 확인했다.
3. 동일 슬롯 예약 경쟁 상황에서 과예약 없이 정합성이 유지되는지 확인했다.
4. 판매자 운영 흐름에서 `READY`, `PICKED_UP`, `NO_SHOW` 상태 전이가 안정적으로 이어지는지 확인했다.
5. 취소 후 재예약 시 재고와 슬롯 잔여 수량이 정상 복구되는지 확인했다.
6. 별도 capacity 보완 실행으로 상품 목록 조회 API의 단계별 동접 수용 구간을 확인했다.

## 해석 기준

- `409 Conflict`는 경합 시나리오에서는 정상적인 비즈니스 충돌 거절로 해석했다.
- `5xx`는 실제 서버 장애로 해석했다.
- `http_req_failed`는 `409`도 포함할 수 있어, 최종 해석은 `check pass rate`, 상태코드 분포, `5xx 여부`를 함께 기준으로 삼았다.

## 시나리오별 결과

### 1. 상품 목록 조회

**결과**

- 총 요청 수: 2250
- 평균 응답시간: 3.04ms
- p95 응답시간: 5.9ms
- 최대 응답시간: 65.28ms
- 실패율: 0%
- Check Pass Rate: 100%

**해석**

- 조회 baseline은 매우 안정적이었다.
- 현재 constrained stack 기준에서 조회 API는 충분한 성능 여유를 보였다.

### 2. 상품 등록 burst

**결과**

- 총 요청 수: 820
- `201 Created`: 140
- `409 Conflict`: 680
- `5xx`: 0
- 평균 응답시간: 12.47ms
- p95 응답시간: 18.56ms

**해석**

- 중복 SKU를 의도적으로 많이 만들었기 때문에 `409` 비중이 높은 것은 설계된 결과다.
- 핵심은 충돌이 서버 오류가 아니라 `409`로 빠르게 처리됐다는 점이다.

### 3. 핫 슬롯 예약 경쟁

**결과**

- 총 요청 수: 907
- `201 Created`: 6
- `409 Conflict`: 445
- `5xx`: 0
- 평균 응답시간: 3.24ms
- p95 응답시간: 5.56ms
- teardown 정합성 검증: 통과

**해석**

- 성공 예약 수가 낮은 것은 경쟁 대상 슬롯의 잔여 정원이 작았기 때문이다.
- 중요한 점은 과예약, 음수 재고, 음수 잔여 정원이 발생하지 않았다는 것이다.

### 4. 판매자 예약 운영 상태 전이

**결과**

- 예약 생성 성공: 100
- `READY` 전이 성공: 50
- `PICKED_UP` 전이 성공: 50
- `NO_SHOW` 전이 성공: 50
- 평균 응답시간: 9.02ms
- p95 응답시간: 16.92ms
- 실패율: 0%

**해석**

- 판매자 목록 조회와 상태 변경이 반복돼도 안정적으로 동작했다.
- 운영 흐름의 두 갈래인 `READY -> PICKED_UP` 과 `NO_SHOW`가 모두 정상 확인됐다.

### 5. 취소 후 재예약

**결과**

- 예약 생성 성공: 100
- 취소 성공: 200
- 재예약 성공: 100
- `409`: 0
- `5xx`: 0
- 평균 응답시간: 8.84ms
- p95 응답시간: 30.28ms

**해석**

- 취소 후 자원 복구가 정상 수행됐다.
- 복구 직후 재예약도 문제없이 이어져, 재고와 슬롯 잔여 수량 관리가 정합성을 유지했다.

## Capacity 보완 결과

**결과**

- 총 요청 수: 85120
- `2xx`: 85120
- `5xx`: 0
- 평균 응답시간: 2.5ms
- p90 응답시간: 3.9ms
- p95 응답시간: 4.82ms
- 최대 응답시간: 715.43ms
- 처리량: 159.33 req/s
- 관찰 구간: `10 -> 25 -> 50 -> 100 -> 150 -> 200 VUs`

**해석**

- 이번 실행 범위에서는 성능 저하 시작 구간과 명확한 한계 구간이 관찰되지 않았다.
- 따라서 현재 constrained stack 기준으로는 상품 목록 조회 API가 최소 `200 VUs 이상`을 수용한다고 해석할 수 있다.
- 실제 한계를 찾으려면 `250/300/400 VUs` 구간까지 확장해야 한다.

## 최종 결론

이번 테스트를 통해 데모앱은 조회, 쓰기 burst, 예약 경쟁, 판매자 운영 상태 전이, 취소 후 재예약 관점에서 기본 안정성을 확보한 상태로 확인됐다. 모든 시나리오에서 `5xx`는 발생하지 않았고, 경합 시나리오의 `409`는 의도된 비즈니스 충돌 제어로 해석할 수 있었다.

추가 보완 실행에서도 조회 capacity는 `200 VUs`까지 유의미한 저하 없이 유지됐다. 따라서 현재 결과는 로컬 baseline 기준의 품질 확인과 회귀 검증 자료로 활용하기에 충분하다. 다만 운영 절대 용량 판단을 위해서는 쓰기 API capacity, 예약 경쟁 capacity, 장시간 soak/stress 테스트가 추가로 필요하다.

## 다음 액션

1. 조회 capacity를 `250/300/400 VUs`로 확장해 실제 한계 구간을 찾는다.
2. 상품 등록 API와 예약 API에 대해서도 step-load capacity 시나리오를 추가한다.
3. 장시간 soak 테스트를 추가해 누적 지연과 자원 누수 가능성을 점검한다.
