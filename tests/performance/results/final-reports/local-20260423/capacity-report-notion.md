# 상품 목록 조회 Capacity 보완 보고서

> 핵심 결론  
> 이번 보완 실행에서는 `10~200 VUs` 전 구간에서 `200 OK`만 관찰됐고 `5xx`는 발생하지 않았다. 현재 constrained baseline 기준으로 상품 목록 조회 API의 수용 한계는 최소 `200 VUs 이상`으로 해석할 수 있다.

## 목적

1차 성능 테스트는 핵심 기능 시나리오의 안정성 확인에 초점이 있었고, 단계별 동접 증가에 따른 수용 구간은 별도로 측정하지 못했다. 이번 보완 실행은 자원 제한 Docker Compose 스택을 기준으로 상품 목록 조회 API의 step-load baseline을 확인하기 위해 수행했다.

## 실행 조건

- 단계: `10, 25, 50, 100, 150, 200 VUs`
- 램프업: 각 단계 20초
- 유지: 각 단계 1분
- 쿨다운: 20초
- 앱, DB, 모니터링: constrained stack
- k6: 호스트 실행

실행 메모:

- 스택 재기동 직후 첫 preflight에서 API warm-up 중 연결 재설정이 한 번 있었다.
- 실제 측정값은 health 확인 후 재실행한 본실행 기준이다.

## 구간별 판단

| 구간 | 판단 | 근거 |
| --- | --- | --- |
| 10~50 VUs | 안정 | 실패와 지연 상승이 관찰되지 않음 |
| 100 VUs | 안정 | 체크 실패 없음, 집계 지표 양호 |
| 150 VUs | 안정 | `5xx` 없음, 처리 유지 |
| 200 VUs | 안정 | 전 구간 `200 OK`, p95 4.82ms |
| 성능 저하 시작 구간 | 미관찰 | p95와 실패율 기준에서 의미 있는 저하 없음 |
| 명확한 한계 구간 | 미관찰 | 200 VUs까지 실패 없이 유지 |

## 해석

- 이번 범위 안에서는 성능 저하 시작 지점이 드러나지 않았다.
- `max 715.43ms` 스파이크는 있었지만 평균과 p95가 매우 낮고 실패가 없어서 지속적 저하로 해석하지 않았다.
- 따라서 이번 결과는 `상품 목록 조회 API가 최소 200 VUs 이상을 안정적으로 처리했다`는 baseline 근거로 사용할 수 있다.

## 결론

이번 보완 실행은 `버틴 최대 VU 수`보다 `어느 구간까지 안정적이었는가`를 보기 위한 테스트였다. 그 기준에서 보면 현재 constrained stack 환경에서 상품 목록 조회 API는 `10~200 VUs` 전 구간을 안정적으로 처리했다. 이번 범위 안에서는 병목 구간이 확인되지 않았으므로, 다음 단계는 더 높은 동접 범위로 한계를 직접 찾는 것이다.
