## [실행 전] 2. 데모앱 목적 및 범위

### 2.1 데모앱 구성 목적

- 본 데모앱은 일반 서비스 전 기능을 구현하기 위한 제품이 아니라, `k6` 기반 성능 테스트 시나리오를 재현하기 위한 최소 기능 데모로 구성했다.
- 검증 대상은 조회 성능, 쓰기 burst, 예약 경합, 판매자 예약 운영 상태 전이, 취소 후 재예약, capacity 같은 성능 테스트 핵심 시나리오다.
- 따라서 본 보고서도 기능 설명 자체보다, 해당 시나리오를 어떤 전제로 검증했는지와 결과를 어떻게 해석할지를 중심으로 작성한다.

### 2.2 최소 구현 범위

- 테스트 시나리오에 직접 필요한 기능만 포함한다.
  - 매장
  - 데모 사용자(판매자/고객)
  - 상품 조회/등록/수정
  - 픽업 슬롯 조회
  - 예약 생성
  - 예약 상태 변경
  - 예약 취소
- 회원가입/실사용 인증, 결제, 장바구니, 검색/추천, 쿠폰/프로모션, 정산, 외부 연동은 본 테스트 목적과 직접 관련이 없으므로 제외했다.
- 이 같은 최소 범위 구성은 성능 병목을 `조회`, `쓰기 충돌`, `재고/슬롯 차감`, `취소 후 복구`, `동접 증가`에 집중해서 해석하기 위한 것이다.

### 2.3 시스템 구성

- Frontend: React + Vite
- Backend: Fastify API
- Database: PostgreSQL
- Monitoring: Prometheus, Grafana
- Load Generator: k6

- 판매자 상품 등록 burst 처리 성능
- 동일 픽업 슬롯 동시 예약 경합 처리
- 판매자 예약 목록 조회 및 상태 변경 처리 성능
- 예약 취소 후 재예약 상태 전이 처리

### 2.5 테스트 제외 범위

- 브라우저 렌더링 성능
- 외부 연동 시스템

## [실행 전] 3. 테스트 환경 및 구성

### 3.1 환경 구성

- 본 보고서의 부하테스트는 `k6`를 로컬 호스트에서 실행하고, 테스트 대상 애플리케이션은 별도 환경으로 구분해 실행하는 방식을 기준으로 한다.
- 로컬 baseline 조건을 만들 때는 `docker-compose.stack.yml`, `docker-compose.monitoring.yml`, `docker-compose.capacity.yml` 조합으로 앱 스택을 띄우고 CPU/메모리 제한을 적용한다.
- 이 경우 테스트 경로는 `로컬 호스트 k6 -> localhost:14000 API -> PostgreSQL/Monitoring 컨테이너` 구조다.
- 개발 편의 목적의 로컬 검증에서는 `npm run dev:api` 기준 `http://localhost:4000` 을 대상으로 동일한 시나리오를 실행할 수 있다.

### 3.2 테스트 구성요소 및 선택 이유

- Frontend
  판매자와 고객 흐름을 수동으로 확인할 수 있는 최소 UI로 구성했다. 본 보고서의 주된 성능 측정 대상은 아니지만, 조회/예약/상태 전이 흐름을 눈으로 검증하는 보조 수단으로 사용한다.
- API
  상품 조회, 상품 등록, 예약 생성, 예약 취소, 상태 변경 등 핵심 비즈니스 로직을 담당한다. 이번 성능 테스트의 주된 측정 대상이며, 응답시간과 처리량, 충돌 제어, 오류 발생 여부를 이 계층 중심으로 해석한다.
- Database
  상품 재고, 픽업 슬롯 잔여 수량, 예약 상태를 저장한다. 특히 예약 경쟁과 취소 후 재예약 시나리오에서 재고 차감/복구와 슬롯 정원 차감/복구를 비교적 현실적으로 확인하기 위해 사용한다.
- Load Generator
  `k6` 는 애플리케이션과 분리된 호스트에서 실행한다. 이렇게 해야 부하 생성기가 앱 컨테이너 자원을 함께 점유하지 않아 측정 해석이 단순해지고, 시나리오를 반복 실행하기도 쉽다.
- Monitoring
  Prometheus와 Grafana를 사용해 `응답시간`, `처리량`, `VUs`, `check pass rate`, `dropped iterations` 를 함께 관찰한다. 단순 콘솔 결과보다 시나리오별 추세와 구간별 변화를 해석하기 쉽다.
- Docker Compose 기반 로컬 스택
  로컬에서도 API, DB, 모니터링 구성을 일정하게 맞추기 위해 사용한다. 필요 시 자원 제한을 걸어 capacity 기준의 baseline 환경을 재현할 수 있다는 점에서 선택했다.

### 3.3 테스트 데이터 및 계정

- 기본 계정
  - 판매자: `seller-seoul-central`
  - 고객: `customer-minji`
  - 매장: `store-seoul-central`
- 테스트 데이터 전제
  - 대상 매장에 조회 가능한 상품이 최소 1개 이상 존재해야 한다.
  - 대상 날짜에 픽업 슬롯이 최소 1개 이상 존재해야 한다.
  - 예약 경쟁 테스트 전에 대상 상품 재고와 슬롯 잔여 수량이 충분해야 한다.

### 3.4 한계

- 본 구성은 로컬에서 자원 제한 baseline을 재현하기 위한 환경이며, 실제 운영 환경 자체를 그대로 복제한 것은 아니다.
- `k6`가 호스트에서 단일 실행기로 동작하므로, 실무에서 사용하는 분산 부하 생성 환경(EKS/EC2 다중 인스턴스, k6 operator, SaaS load generator)의 네트워크 특성과 source IP 분산은 반영하지 않는다.
- 단일 호스트, 단일 Docker Compose 스택 기준이므로 로드밸런서, 오토스케일링, 멀티 AZ, 멀티 인스턴스, 서비스메시, WAF 같은 운영 계층의 병목이나 보호 동작은 검증하지 못한다.
- 따라서 본 결과는 `고정 자원 예산 내 상대 비교`, `병목 징후 탐지`, `회귀 확인`에는 유효하지만, 실제 운영 최대 처리량이나 절대 수용량을 단정하는 근거로는 제한이 있다.

### 4.3 시나리오 및 스크립트 설명

#### 상품 목록 조회

- 스크립트: `tests/performance/k6/product-list-read.js`
- 동작 방식:
  - 각 VU가 같은 매장의 상품 목록 API를 반복 호출한다.
  - 응답이 `200` 인지, 응답 본문이 배열 형태인지 확인한다.
  - 각 반복 사이에 짧은 `sleep` 을 두어 완전 무간격 폭주가 아니라 실제 조회 트래픽에 가까운 패턴으로 본다.
- 이 스크립트로 확인하는 것:
  - 읽기 중심 API의 기본 응답시간 안정성
  - 동일 조회가 반복될 때 지연시간 분산이 커지는지 여부
  - 조회 API가 가장 기본적인 정상 응답 형태를 유지하는지 여부
- `constant-vus` 를 쓴 이유:
  - 조회 시나리오는 급격한 사용자 증가보다 일정한 동시 요청 하에서 안정적으로 버티는지가 더 중요하기 때문이다.

#### 상품 등록 burst

- 스크립트: `tests/performance/k6/product-registration-burst.js`
- 동작 방식:
  - 판매자 권한으로 상품 등록 API를 반복 호출한다.
  - SKU 풀을 의도적으로 좁게 잡아 중복 SKU 충돌이 발생하도록 만든다.
  - 응답 상태를 `201`, `409`, `5xx` 로 구분해 카운트한다.
- 이 스크립트로 확인하는 것:
  - 쓰기 부하가 증가할 때 상품 등록 처리량이 어떻게 변하는지
  - SKU 충돌이 서버 오류가 아니라 `409` 로 제어되는지
  - 권한 오류나 `5xx` 같은 비정상 응답이 섞이지 않는지
- `ramping-vus` 를 쓴 이유:
  - 쓰기 요청이 점진적으로 몰릴 때 충돌 비율과 응답시간이 어떻게 바뀌는지 보기 위해서다.

#### 핫 슬롯 예약 경쟁

- 스크립트: `tests/performance/k6/hot-slot-race.js`
- 동작 방식:
  - `setup()` 단계에서 테스트 전용 상품 하나를 만들고, 잔여 수량이 남은 픽업 슬롯 하나를 선택한다.
  - 실행 중에는 여러 고객 계정이 같은 슬롯을 대상으로 예약 요청을 보낸다.
  - 슬롯 조회가 가능한지 먼저 확인한 뒤 예약 생성 API를 호출하고, 응답 상태를 `201`, `409`, `5xx` 로 구분한다.
- 이 스크립트로 확인하는 것:
  - 동일 상품과 동일 슬롯에 동시 요청이 몰릴 때 과예약 없이 충돌을 제어하는지
  - 재고 차감과 슬롯 정원 차감이 동시에 경합되는 상황에서 서버 오류 없이 동작하는지
  - 성공 예약과 정상 거절이 어떤 비율로 분포하는지
- `constant-arrival-rate` 를 쓴 이유:
  - 응답속도와 무관하게 초당 요청 유입량을 일정하게 유지해야 실제 경쟁 상황을 더 명확히 재현할 수 있기 때문이다.

#### 판매자 예약 운영 상태 전이

- 스크립트: `tests/performance/k6/reservation-status-flow.js`
- 동작 방식:
  - `setup()` 단계에서 VU별 전용 상품과 슬롯 날짜를 준비한다.
  - 각 반복에서 고객이 예약을 생성한 뒤, 판매자가 같은 날짜의 예약 목록을 조회한다.
  - 이후 반복마다 분기를 나눠 일부 예약은 `READY -> PICKED_UP`, 일부 예약은 `NO_SHOW` 로 상태 변경한다.
  - 상태 변경 후 예약 목록에서 최종 상태가 반영되었는지 확인한다.
- 이 스크립트로 확인하는 것:
  - 판매자 예약 운영 API의 응답 안정성
  - 예약 생성 직후 후속 운영 흐름이 정상적으로 이어지는지
  - 픽업 완료 분기와 미수령 분기가 함께 반복될 때 `5xx` 없이 동작하는지
- `per-vu-iterations` 를 쓴 이유:
  - 각 VU가 자기 전용 상품과 날짜를 기준으로 같은 운영 흐름을 반복해야 목록 조회와 상태 전이를 비교적 일관되게 검증할 수 있기 때문이다.

#### 취소 후 재예약

- 스크립트: `tests/performance/k6/cancel-and-rereserve.js`
- 동작 방식:
  - `setup()` 단계에서 VU별 전용 상품과 슬롯 날짜를 따로 준비한다.
  - 각 반복에서 예약 생성 후 재고와 슬롯 잔여 수량이 감소했는지 확인한다.
  - 이어서 예약 취소 후 자원이 원복되었는지 확인하고, 다시 동일 조건으로 재예약을 시도한다.
  - 생성, 취소, 재예약, 최종 취소의 각 단계 상태를 점검한다.
- 이 스크립트로 확인하는 것:
  - 상태 전이 흐름이 정상적으로 이어지는지
  - 취소 직후 재고와 슬롯 수량이 정확히 복구되는지
  - 복구 이후 후속 재예약이 정상적으로 가능한지
- `per-vu-iterations` 를 쓴 이유:
  - 각 VU가 자기 전용 자원으로 같은 흐름을 반복해야 생성-취소-재예약 과정을 비교적 결정론적으로 검증할 수 있기 때문이다.

### 4.4 측정 지표

- 총 요청 수
- 초당 요청 수
- 평균 응답시간
- p90 응답시간
- p95 응답시간
- 최대 응답시간
- 실패율
- 상태코드 분포
- 본 성능테스트는 단순히 평균 응답시간만 보는 것이 아니라, `응답시간`, `처리량`, `체크 통과율`, `상태코드 분포`, `동시 사용자 변화`, `dropped iterations` 를 함께 본다.
- 응답시간은 `avg`, `p90`, `p95`, `max` 를 기준으로 확인한다.
- 처리량은 `총 요청 수`, `HTTP Requests / s`, `Iterations / s` 로 확인한다.
- 안정성은 `check pass rate`, `http_req_failed`, `dropped iterations`, `5xx 발생 여부` 로 확인한다.
- 따라서 본 보고서의 핵심 판단 기준은 `느려졌는가` 뿐 아니라 `정상 응답을 유지했는가`, `비즈니스 충돌을 올바르게 처리했는가`, `실제 장애 없이 버텼는가` 이다.

### 4.6 상태코드 및 결과 해석 기준

| 상태코드 | 의미 | 이 프로젝트에서의 대표 사례 | 보고서 해석 |
| --- | --- | --- | --- |
| `200 OK` | 요청 정상 처리 | 목록 조회 성공, 예약 취소 성공 | 정상 성공 |
| `201 Created` | 자원 생성 성공 | 상품 등록 성공, 예약 생성 성공 | 정상 성공 |
| `400 Bad Request` | 요청 값 오류 | 필수 필드 누락, 잘못된 파라미터 | 테스트 입력 또는 검증 로직 문제 |
| `401 Unauthorized` | 인증 정보 없음 | `x-user-id` 누락 | 테스트 설정 오류 |
| `403 Forbidden` | 권한 없음 | 고객이 판매자 API 호출 | 권한 제어 검증 포인트 |
| `404 Not Found` | 대상 리소스 없음 | 없는 상품, 예약, 슬롯 조회 | 데이터 준비 문제 가능성 |
| `409 Conflict` | 현재 상태와 충돌 | 중복 SKU, 재고 부족, 슬롯 마감, 동시 예약 충돌 | 경합 시나리오에서는 정상 거절일 수 있음 |
| `500 Internal Server Error` | 서버 내부 오류 | 예외 처리 실패, DB 처리 오류 | 실제 장애로 봐야 함 |

해석 원칙:
- `200`, `201`은 비즈니스 성공으로 본다.
- `409`는 서버 장애가 아니라 비즈니스 규칙에 따른 충돌 거절일 수 있다.
- `500`은 실제 시스템 장애로 본다.
- `http_req_failed`는 `4xx`도 포함할 수 있으므로, 결과 해석 시 `check 통과율`과 상태코드 분포를 함께 기록한다.

## [실행 후] 9. 기승전결 서술 예시

### 9.1 최종보고서 본문 예시

#### 기

본 성능 테스트는 `k6 Pickup Market Demo` 애플리케이션을 대상으로, 조회 성능뿐 아니라 쓰기 burst, 예약 경합, 취소 후 재예약 같은 실제 서비스 운영 중 병목과 장애로 이어질 수 있는 구간을 사전에 확인하기 위해 수행하였다. 테스트는 로컬 3-tier 환경을 기준으로 `k6` 를 호스트에서 실행하고, API와 PostgreSQL 및 모니터링 스택은 별도 프로세스 또는 컨테이너로 분리해 구성하였다. 다만 본 결과는 단일 호스트와 단일 load generator 기반의 baseline 측정이므로, 실제 운영 환경의 절대 성능 보증치라기보다 회귀 검증과 병목 징후 탐지 기준으로 해석하는 것이 적절하다.

#### 승

이번 테스트에서는 총 다섯 가지 핵심 시나리오를 검증하였다. 첫째, 상품 목록 조회 시나리오를 통해 반복 조회 트래픽에서 응답시간이 안정적으로 유지되는지 확인하였다. 둘째, 상품 등록 burst 시나리오를 통해 동일 매장으로 쓰기 요청이 집중될 때 처리량과 SKU 충돌 제어가 정상적으로 동작하는지 점검하였다. 셋째, 핫 슬롯 예약 경쟁 시나리오를 통해 동일 상품과 동일 픽업 슬롯에 동시 요청이 몰리는 상황에서 과예약 없이 충돌을 제어할 수 있는지 확인하였다. 넷째, 판매자 예약 운영 상태 전이 시나리오를 통해 예약 목록 조회와 `READY`, `PICKED_UP`, `NO_SHOW` 상태 변경 흐름이 안정적으로 이어지는지 검증하였다. 다섯째, 취소 후 재예약 시나리오를 통해 예약 생성 이후 취소, 자원 복구, 후속 재예약까지의 상태 전이가 정상적으로 이어지는지 확인하였다.

테스트 결과 해석에는 평균 응답시간만 사용하지 않았다. `avg`, `p90`, `p95`, `max` 응답시간과 함께 `HTTP Requests / s`, `Iterations / s`, `Check Pass Rate`, 상태코드 분포, `Dropped Iterations` 를 종합적으로 확인하였다. 특히 `409 Conflict` 는 경합 상황에서의 정상 거절일 수 있으므로 비즈니스 규칙에 따른 제어 결과로 해석하고, `5xx` 응답은 실제 서버 오류로 간주하는 기준을 적용하였다.

#### 전

테스트 결과, 상품 목록 조회 시나리오에서는 `[기입: 총 요청 수]` 건의 요청 동안 평균 응답시간 `[기입]`, `p95` 응답시간 `[기입]` 으로 확인되어 조회 트래픽에 대한 기본 응답 안정성을 평가할 수 있었다. 상품 등록 burst 시나리오에서는 `201` 응답 비율 `[기입]`, `409` 응답 비율 `[기입]` 으로 나타났으며, 이는 중복 SKU 또는 충돌 상황이 시스템 오류가 아니라 의도된 비즈니스 제어로 처리되었는지 판단하는 근거가 된다. 핫 슬롯 예약 경쟁 시나리오에서는 `201` 과 `409` 응답 분포 및 `5xx` 발생 여부를 통해 과예약 없이 경합이 제어되었는지 확인하였다. 판매자 예약 운영 상태 전이 시나리오에서는 예약 목록 조회 성공 여부와 `READY`, `PICKED_UP`, `NO_SHOW` 전이 성공 수를 통해 판매자 운영 API가 안정적으로 이어지는지 확인하였다. 취소 후 재예약 시나리오에서는 생성 성공 수 `[기입]`, 취소 성공 수 `[기입]`, 재예약 성공 수 `[기입]` 를 기준으로 재고 및 슬롯 정원이 정상적으로 복구되고 재사용 가능한지 검증하였다.

추가로 capacity 보완 테스트에서는 단계별 동접 증가에 따라 응답시간 상승 구간과 장애 발생 구간을 분리해 해석할 수 있었다. `[기입: 안정 운영 가능 구간]` 까지는 `200` 응답과 체크 통과율이 비교적 안정적으로 유지되었고, `[기입: 성능 저하 시작 구간]` 부터는 `p95` 응답시간 증가 또는 체크 실패 증가가 관찰되었다. `[기입: 명확한 한계 구간]` 에서는 `5xx` 또는 유의미한 성능 저하가 발생해 현재 baseline에서의 한계 구간으로 판단하였다.

#### 결

종합하면, 이번 성능 테스트를 통해 `k6 Pickup Market Demo` 는 조회, 쓰기 burst, 예약 경합, 상태 전이, capacity 관점에서 핵심 동작을 검증할 수 있었고, 현재 로컬 baseline 환경에서 안정 운영 가능 구간과 성능 저하 시작 구간을 구분할 수 있었다. 특히 본 테스트는 단순히 "빠른가"를 보는 것이 아니라, 충돌 상황에서 시스템이 올바르게 거절하는지, 과예약 없이 정합성을 유지하는지, 실제 장애 없이 요청을 처리하는지를 함께 확인했다는 점에 의미가 있다. 다만 운영 환경의 절대 수용량을 판단하려면 분산 load generator, 운영 유사 인프라, 장시간 soak/stress 테스트가 추가로 필요하다. 따라서 본 보고서의 결과는 현 시점에서의 로컬 baseline 품질 확인과 회귀 검증 기준으로 활용하는 것이 가장 합리적이다.

### 9.2 짧은 결론 예시

이번 테스트를 통해 `k6 Pickup Market Demo` 의 조회 성능, 상품 등록 burst 처리, 예약 경합 제어, 판매자 예약 운영 상태 전이, 취소 후 재예약 정합성, 단계별 capacity 특성을 확인하였다. 결과 해석은 응답시간뿐 아니라 처리량, 체크 통과율, 상태코드 분포, `Dropped Iterations` 를 함께 기준으로 삼았으며, `409` 는 정상 충돌 제어로, `5xx` 는 실제 장애로 분리해 판단하였다. 그 결과 현재 로컬 baseline 환경에서 안정 운영 가능 구간과 성능 저하 시작 구간을 구분할 수 있었고, 향후 threshold 정교화와 장시간 부하 테스트를 통해 운영 유사 기준을 더 보완할 필요가 있다.
